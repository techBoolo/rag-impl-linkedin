# FAISS Vector Store Integration

This guide covers how we integrated an in-memory **FAISS (Facebook AI Similarity Search)** vector store to hold the embeddings generated by our RAG pipeline.

## 1. Overview
FAISS allows us to efficiently search for embeddings that are similar to a query embedding. By storing our document chunks and their embeddings in FAISS, we set the foundational stage for the "Retrieval" part of our application.

## 2. Implementation

### Dependency
We added `faiss-cpu` to the environment using `uv`:
```bash
uv add faiss-cpu
```

### The `create_faiss_index` Function
We added an asynchronous function `create_faiss_index` in `main.py` that handles both the initial creation and appending of embeddings dynamically:
- Takes a batch of text chunks and their corresponding embedding vectors.
- Packages texts with their respective vectors and metadata.
- Initializes a new FAISS vector store using the `FAISS.afrom_embeddings` constructor if one doesn't exist yet.
- Progressively appends to the store using `vector_store.add_embeddings` for subsequent batches.

### Integrating into the Main Pipeline
In the `main()` function, we updated our iterative loop:
- Created a persistent `vector_store = None`.
- Yielded batches of embeddings from `process_embeddings`.
- Passed the generated batch sequences to `create_faiss_index` iteratively to build our index without exceeding memory loads.
- Printed final statistics showing successful ingestion.

## 3. Results
Running the script generates the following output:
```text
Processed batch 11 (10 chunks) -> Added to FAISS vector store. Total chunks embedded: 110
Processed batch 12 (4 chunks) -> Added to FAISS vector store. Total chunks embedded: 114

Successfully created FAISS index with 114 chunks across 12 batches!
Final vector store doc count: 114
```
This confirms that our in-memory FAISS store successfully aggregated all the document embeddings!
